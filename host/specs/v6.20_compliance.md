# v6.20 SPU Firmware — Empirical Spec Compliance Baseline

**Generated**: 2026-04-27 (Phase 0.2)

## TL;DR

The v6.20 SPU firmware (released 2016-08-18) implements **MAN BWM specification revision 05 (informally "8.5", dated 2012-09-27)**, plus four years of post-2012 field-fix accumulation that does *not* trace back to any newer revision of the spec. Specifically:

- Every spec 8.5 algorithmic requirement we can identify has a corresponding firmware site.
- Several spec 8.6+ requirements (rev 06, 2017-11-27) are absent — they post-date the firmware.
- The firmware contains one significant addition that is in *no* spec version: an MB-sum (main-bearing neighbour-sum) alarm pathway. This is custom field-driven work.
- A handful of partial implementations exist where v6.20 has the spec 8.5 mechanism but with constants or behavior that the team adjusted in service.

The full row-by-row matrix is in `feature_matrix.csv` (66 atomic requirements). This document explains the headline conclusions and lists the 5 strongest pieces of evidence that v6.20 == 8.5.

---

## The five strongest pieces of evidence v6.20 implements spec 8.5

### 1. Dual-formula dynamic alarm with `Alarm_1` and `Alarm_2`

Spec 8.5 §4.5 specifies:

```
Alarm_Dynamic = max(Alarm_1, Alarm_2)
Alarm_1       = Alarm_Basic · (1 + ΔRPM_increment / k)
Alarm_2       = Alarm_Basic · (1 + ΔRPM_increase_at_alarm / k) · exp(-Hits/β)
```

Spec 8.6/8.7 §4.4 collapsed this to:

```
Alarm_Dynamic = Alarm_Basic · (1 + ΔRPM_PCT/k),  k=16   /* no β, no exp, no max */
```

The firmware's `dynalarmlevels` struct (`man.c:808–849`) carries fields named `single1`, `single2`, `cyl1`, `cyl2`, `mbsum1`, `mbsum2` plus their `_int` rounded variants. The "1" / "2" suffixes correspond directly to spec 8.5's `Alarm_1` and `Alarm_2`. The function `CalcAlarmLevel1` is invoked at `man.c:1328–1333` against `damagemon.alarmlevelsingle/cyl/mbsum`. **This is the single strongest piece of evidence and on its own settles the version question.** No 8.6+ firmware would carry both `single1` and `single2` — it would have one alarm field per metric.

### 2. SingleRapid is `abs(Rapid Wear)`, not `min(|RWS|, |dRS|)`

Spec 8.5 §4.4.1: `Rapid Single = abs(Rapid Wear)`.
Spec 8.6 §4.3.1: `SingleRapid_n = min(|RWS_n|, |dRS_n|)` — uses the new "Rapid Wear deviation" introduced in §4.2 of rev 06.

There is no "Rapid Wear deviation" calculation in v6.20 (no `dRS`, `dRC`, `dRT`). v6.20's SingleRapid is computed as a plain absolute value of the rapid-wear-sensor difference. So v6.20 cannot be implementing 8.6+ SingleRapid; it implements 8.5's.

### 3. TwinCylRapid uses 2-sensor sum, not 4-sensor average

Spec 8.5 §4.4.3: `TwinCylRapid = abs(RW_AftCyl,n + RW_FoCyl,n+1)` (just the two adjacent sensors across cylinders).
Spec 8.6/8.7 §4.3.3: `TwinCylinderRapidAve = min(|RWT|, |dRT|)` where `RWT = (RWS_FoCyl,m + RWS_AftCyl,m + RWS_FoCyl,m+1 + RWS_AftCyl,m+1)/4` (all 4 sensors, averaged).

v6.20's twin-cylinder calc takes 2 sensors. It does not take 4-sensor averages.

### 4. `man_comptable[150][28]` confirms the "≥100 sub-ranges" requirement of spec 2.0

`man.c:50` declares the speed-compensation lookup as `int man_comptable[150][28]` — 150 RPM rows, 28 sensor channels. Spec §2.0 (all versions) specifies "no less than 100 sub-ranges". v6.20 ships 150 — exceeding the spec floor. The 28-channel dimension matches the dual-SPU layout (14 per SPU × 2). This is consistent across 8.5/8.6/8.7, so it doesn't *distinguish* version, but it does *confirm* the firmware was built to spec.

### 5. Sensor adjustment is one combined procedure

Spec 8.5 §5 ("Sensor Adjustment") describes a single 50-hour procedure that combines offset re-adjustment with cylinder-neighbour wear extrapolation, using the formula `O = O₁ + a₁·t_b + 500·a₃` (three RMS lines).

Spec 8.6 §5 ("Sensor Failure") splits this into 5.1 corrections-for-failed-sensors plus 5.2 with **three** separate procedures (5.2.1, 5.2.2, 5.2.3) plus 5.2.4 further-actions.

v6.20 has one sensor-replacement workflow (the `iSensorInRepalce` / `iSensorInRepalceMode` / `iSensorInRepalceTimer` flags in `man.c`, with the entry point in `PCModBus.c:1059–1065`). It does not have three branched procedures.

## Summary of spec-8.6+ requirements *missing* from v6.20

These are the items the project's Phase 4 (upgrade to 8.7) must implement. All come from rev 06; rev 07 adds nothing new beyond `Rapid Slow = 0` in §4.5 and §5.2.4.

### Algorithm changes
1. **Rapid Wear deviation calculations** (§4.2) — the `dRS`, `dRC`, `dRT` formulas. Required to support new `min(|RWS|, |dRS|)`-style alarm composition.
2. **SingleRapid / CylinderRapidAve / TwinCylinderRapidAve renamed and re-defined** (§4.3.x). Math change, not just naming.
3. **TwinCylinderRapidAve uses 4-sensor average** (§4.3.3) — was 2-sensor sum in 8.5.
4. **Dynamic alarm formula simplification** (§4.4) — drop `exp(-Hits/β)`, drop the max(), use single formula with k=16.
5. **ΔRPM gating in % of nominal**, not absolute rpm (§4.4) — was "if ΔRPM > 4 rpm cap to 4 rpm", now "if 16% ≤ ΔRPM% ≤ 25% set ΔRPM_PCT=16; if > 25% pause monitoring".
6. **Reset on compensation curve update**: `Rapid Slow = 0` (§4.5, only in 8.7) vs. the 8.6/8.5 expressions.
7. **Sample stability check during refining** (§2.2): discard sample if change > 5% nominal between consecutive samples.
8. **Final compensation curve weighted blend** (§2.3): `REF = (Nhits·V_avg + (1000-Nhits)·V_interp)/1000` for under-sampled speed points.
9. **Three sensor-adjustment procedures** (§5.2.1 / 5.2.2 / 5.2.3) replacing one combined procedure.
10. **Discard barred-speed-band data in rapid wear** (§4) — verify v6.20's barred-speed handling already covers this.

### Storage requirements (entirely new in 8.6)
11. **Hit-by-hit data of uncompensated signals** (§6.2 + Appendix E): 12h of data, 1 line per revolution, with explicit text format. Extended +1h after rapid-wear alarm, +1h after slow-wear alarm release.
12. **Compensation-curve change log** (§6.2 + Appendix F): timestamped file each time speed-comp curves are updated, with explicit text format.
13. **Rapid Wear field list in 2h short-term storage** (§6.3): explicit field set: `timestamp, rpm, Rapid wear sensor, Rapid fast, Rapid slow, SingleRapid, CylRapidAve, TwinCylAve, deltaRpm, Rapid Alarm`.
14. **Frozen copy retention policy**: ≥1 year; 65h hit-by-hit always available (corresponds to 5 alarm events).
15. **Long-term storage extends 5 years back** (§6.4) — verify v6.20 storage size.
16. **Event log content**: 5-item list (alarms/slowdowns/pre-warnings, replacement/offset, rough→fine cal change, pre-warning ref reset, all timestamped). Verify v6.20 log format covers all five.

### Interface (entirely new chapter 8 in 8.6)
17. **Alarm relay XS8126**, normally closed, toggle on level escalation, 0.1–5s adjustable toggle period (default 1s).
18. **Slowdown relay XS8127**, normally open, with supervision resistor terminals.
19. **System Fail relay XS8129**, normally closed, triggered on power fail / watchdog / monitor-unavailable.
20. **XML serialization format** (Appendices A-D) for class surveyor presentation.

### Alarm limit values (tighter in 8.6)
21. SingleRapid: 250 µm → **200 µm**.
22. CylRapidAve: 180 µm → **90 µm** (-50%).
23. TwinCylRapidAve: 250 µm → **75 µm** (-70%).
24. Three separate alarm tables (normal running / refining / rough creation) replacing 8.5's single set.
25. Slow Wear slowdown limits: sensor-deviation in normal running 0.5 mm → 0.6 mm; in learning 0.7 mm → 0.5 mm.

## Things v6.20 has that are *not* in any spec

These are field-driven additions that exist because the team needed them in production. They have to be either preserved through the refactor (if still operationally needed) or carefully retired (if obsoleted by spec 8.6+).

| Feature | File:line | Rationale (best guess) |
|---|---|---|
| **MB-sum alarm pathway** (`mbsum1`, `mbsum2`, `alarmlevelmbsum`) | `man.c:810–849, 1332` | An additional combined-alarm metric for main-bearing neighbours. Likely added because the cylinder/twin-cyl algorithm did not give clear signals for main-bearing wear in some real-world scenarios. **Not** in spec 8.5 / 8.6 / 8.7. |
| **CPP / sensor moving / SLEM / DC mode** features | various | Product features added in v6.10–v6.19 changelog (`version.c` release notes 2012–2015). These are SPU product features layered on top of the BWM spec, likely tied to specific customer installations. |
| `damagemonignore[]` 3-cycle ignore counter | `man.c:776, 787, 1135` | Mechanism to suppress damage-monitoring output for the first few cycles after a sensor replacement / FRAM reload. Pragmatic field hack. Possibly the source of Bug 1 (sensor replacement zeroes other sensors' RPM). |

## Methodology notes

- The matrix in `feature_matrix.csv` is a manual cross-reference, not an automated tool. Each row was authored from the cleaned spec text at `host/specs/extracted/{8.5,8.6,8.7}/*.md` plus targeted greps in the firmware tree.
- "PARTIAL" entries indicate v6.20 has *something* related but not exactly matching the spec. Phase 1 unit tests will surface the actual behavior, at which point we either reclassify to "Y" (matches once we see the test pass) or "N" (matches confirms the gap).
- "NOT FOUND" entries are the gap we will close in Phase 4. They are also opportunities for "is this in the C# CPU dashboard, not the SPU firmware?" — for storage requirements (XML output, surveyor presentation) the CPU side is a likely home.

---

## What to do with this baseline

1. **Phase 1 (test coverage)** anchors against the "Y" entries with `present_in_8.5=Y`. Each Y row should produce ≥1 unit test. Tests use `@spec 8.5 §X.Y` tags.

2. **Phase 4 (upgrade to 8.7)** picks up the "Missing in v6.20" list above (items 1–25). Each becomes an implementation ticket, gated on a failing `@spec 8.7 §X.Y` test that goes green when the feature lands.

3. **Phase 5 (documentation)** uses `feature_matrix.csv` plus `coverage_matrix.md` (auto-generated) to produce the customer-facing `host/spec_8.7_compliance.md`. The "Things v6.20 has that are not in any spec" section above tells us what extra documentation we need beyond the spec — these are field-driven features that need their own design memos.
